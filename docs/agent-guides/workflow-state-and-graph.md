# Workflow State And Graph

## Shared State Contract

`BIState` in `src/bi_copilot/graph/state.py` is the central workflow state. Keep agent inputs and outputs aligned with this class.

Important fields:

- `user_question`: original user question
- `intent`: planner-derived intent label
- `plan`: full planner output as a dict
- `retrieval_targets`: planner-selected evidence sources
- `retrieved_context`: retrieved metadata/document/context payloads
- `database_schema`: all known tables/views mapped to column lists
- `relevant_tables`: planner-selected subset of tables for the question
- `metric_definitions`: governed metric metadata
- `generated_sql`: SQL generated by `SQLAgent`
- `sql_validation_result`: validator output
- `query_result`: SQL result rows
- `analysis_summary`: generated answer/summary
- `chart_spec`: optional chart specification
- `verification_result`: verifier output
- `final_answer`: final user-facing answer
- `errors`: accumulated errors
- `audit_trace`: agent/tool audit events

Important distinction:

- `database_schema` means all available schema context.
- `relevant_tables` means the planner-selected subset after understanding the question.
- Do not feed `relevant_tables` into the planner as if it were all available tables.

## Workflow Implementation Order

When continuing workflow build-out:

1. Keep `BIState` stable and explicit.
2. Implement metadata retrieval node to populate `database_schema`, `metric_definitions`, and table descriptions/context.
3. Wire `PlannerAgent` as the planner node.
4. Wire `SQLAgent` as the SQL generator node.
5. Add SQL validator node using `validate_sql`.
6. Add SQL executor node using `run_readonly_sql`, only after validation passes.
7. Wire `ReporterAgent`.
8. Wire `VerifierAgent`.
9. Build routing in `workflow.py`.
10. Add workflow-level tests with mocked LLMs and database/tool calls.

When implementing workflow behavior, prefer wiring existing agent classes instead of duplicating prompt or parsing logic inside graph nodes.
